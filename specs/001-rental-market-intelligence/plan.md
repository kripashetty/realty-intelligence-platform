## Summary

A Berlin landlord submits their apartment's characteristics via the React frontend. The system
matches those details against a corpus of imported rental listings (uploaded as CSV from Fredy)
and returns a statistically-grounded recommended listing price, a confidence range, a competitive
percentile rank, and a plain-language explanation citing the 3 key market factors driving the
recommendation. The explanation is generated by the Claude API using only the real comparable
data passed in the prompt.

Technical approach: FastAPI (Python 3.11) backend with async SQLAlchemy on Azure PostgreSQL.
Address geocoding via Nominatim in a background task post-import. Comparable filtering via
bounding-box SQL + Python Haversine. Statistical analysis with numpy/scipy. LLM explanation
via Ollama (self-hosted sidecar, llama3.1:8b, CPU for MVP). React 18 + TypeScript frontend
with TanStack Query.

## Technical Context

**Language/Version**: Python 3.11 (backend), TypeScript 5.x strict mode (frontend)

**Primary Dependencies**:
- Backend: FastAPI 0.110+, SQLAlchemy 2.x (async), asyncpg, pandas, numpy, scipy, geopy,
  httpx (Ollama HTTP client), pydantic v2
- Frontend: React 18, TanStack Query v5, React Hook Form, Vite
- Infrastructure: Ollama sidecar container (llama3.1:8b); CPU for MVP, GPU-ready via workload
  profile swap

**Storage**: Azure Database for PostgreSQL Flexible Server 15+

**Testing**: pytest + httpx (backend), Vitest + React Testing Library (frontend)

**Target Platform**: Azure Container Apps (backend), Azure Static Web Apps (frontend)

**Project Type**: Web service (FastAPI) + Web application (React TypeScript)

**Performance Goals**:
- Pricing recommendation returned within 60 seconds of form submission
- CSV import of 1,000 listings processed (excluding geocoding) in under 30 seconds
- Data freshness indicator updates without page reload after upload

**Constraints**:
- TDD mandatory: tests written and failing before implementation (Constitution II)
- All endpoints documented in OpenAPI before UI built against them (Constitution I)
- Business logic in FastAPI services only; React components are display-only (Constitution IV)
- No secrets committed; Key Vault for all credentials (Constitution V)
- Nominatim rate limit: 1 geocoding request/second; geocoding runs as background task
- Ollama CPU inference: ~10–20s per explanation; fits within 60s SLA; GPU upgrade is
  infrastructure-only (no code changes)

**Scale/Scope**:
- MVP: single operator, ~1k–10k listings, Berlin only
- No user authentication in US1; apartment profile is stateless (submitted per request)
- No persistent user accounts; pricing_recommendations stored for audit only
